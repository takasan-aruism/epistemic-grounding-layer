# SPEC S0.2 — 観測ビュー(status_views.py)

task_id: S0-2-status-views
注意: 本 spec は S0.1 の**全文を含む後継版**である。status_views.py は
      丸ごと書き直される。S0/S0.1 の機能を落とさないこと。
S0.2 の変更点は 3 つ:
  (1) runner v0.2 の新イベント形式(events/ 配下)に対応。
      **旧 run_log.jsonl 形式との後方互換を維持**(既存 run が読めなくなる
      ことは退行とみなす)。
  (2) LEDGER の識別子キーに `design_evidence_id` を追加(実データに合わせる)。
  (3) `unparsed` の定義を一本化(JSON 解析失敗のみ)。識別子を持たない行は
      別カウント `ledger_unidentified` とする。
根拠: roadmap-wiring-v0.1 STAGE 0 / TASK-B 実測(DE-0447)
実装ファイル: status_views.py(これのみ。テストファイルは出力禁止)
合格条件: 同梱 test_status_views.py 全件合格(runner が実測する)

## 目的

2DER の一次記録から観測ビューを**導出**する。書かれるものではなく導出される
もの(handoff 置換の第一歩)。LLM を使わない。純粋な読み取りと整形のみ。

## 制約

```
- 標準ライブラリのみ(外部依存禁止)。
- ファイル書き込みをしない。すべての関数は文字列/データを返すだけ。
- 壊れた行で例外を投げて全体を止めない。数えて報告する(可視化された失敗)。
  ただし設定不備・入力パス不在は ViewError。
- 出力順序は決定的(同じ入力なら同じ出力)。
```

## 実装するもの

```python
class ViewError(Exception): ...
```

### 1. load_runs(runs_root) -> list[dict]

`runs_root` 直下の各 run ディレクトリ(`<run_id>/`)を走査。
`RESULT_PACKET.json` と**イベント記録**を読む。

イベント記録の所在は 2 形式ある。**新形式を優先し、無ければ旧形式**を読む:

```
新形式(runner v0.2): <run_id>/events/seg-000001.jsonl
旧形式(runner v0.1): <run_id>/run_log.jsonl
両方あれば新形式を採用する。どちらも無ければ events=0, chain_ok=False,
chain_detail="no event record" とする(証拠なし = ok ではない)。
```

戻り値の各要素:
```python
{"run_id": str, "task_id": str, "status": str, "iterations_used": int,
 "needs_human": bool, "chain_ok": bool, "chain_detail": str,
 "iterations": [ {"iteration": int, "test_exit_code": int|None,
                  "output_tail": str, "reply_excerpt": str} ],
 "events": int}          # run_log の行数
```
- `chain_ok` は hash chain を**自前で再検証**した結果。
  検証の骨格は両形式で同一。**参照するキー名だけが異なる**:

  ```
  旧形式: prev_hash              / seq
  新形式: previous_event_hash    / sequence_in_segment
  ```

  検証規則(この通りに実装すること):
  ```
  prev = "0" * 64      # 先頭行の期待値(genesis)
  seq  = 0             # 先頭行の期待値(0 始まりの連番)
  各行について:
      rec = json.loads(line)
      claimed = rec["record_hash"]
      rec[<prev キー>] != prev  -> 不一致 "line {i}: prev"
      rec[<seq キー>]  != seq   -> 不一致 "line {i}: seq"
      body = {k: v for k, v in rec.items() if k != "record_hash"}
      sha256(json.dumps(body, sort_keys=True, ensure_ascii=False)
             .encode()) != claimed  -> 不一致 "line {i}: hash"
      prev, seq = claimed, seq + 1
  ```
  `json.dumps` の区切り文字は既定のまま(`", "` / `": "`)を使う。
  空行は行として数えない。検証成功なら `chain_detail="ok"`。
  不一致なら `chain_ok=False`、`chain_detail` に最初の不一致位置(例 `"line 3: seq"`)。
  検証成功なら `chain_detail="ok"`。
- `iterations` はイベントから iteration で突合して作る。
  参照するフィールドの所在が形式で異なる:

  ```
  旧形式: レコード直下に kind / iteration / exit_code / output_tail / reply
          kind == "TEST_RUN"    -> exit_code, output_tail
          kind == "MODEL_REPLY" -> reply
  新形式: kind はレコード直下、値は payload の中
          kind == "WORKCELL_TEST_RUN"    -> payload.exit_code,
                                            payload.output_tail
          kind == "WORKCELL_MODEL_REPLY" -> payload.reply
          iteration も payload.iteration を見る
  ```
  `reply_excerpt` は reply の先頭 200 文字。
  TEST_RUN が無い iteration も MODEL_REPLY があれば含める(test_exit_code=None)。
- `events` はイベント記録の行数(新旧いずれか採用した方)。
- RESULT_PACKET.json が壊れている/無い run は結果に含めるが
  `status="UNREADABLE"`, `iterations_used=0`, `needs_human=True` とする。
- 並び順: `run_id` の降順。
- `runs_root` が存在しない場合は空リスト(例外にしない)。

### 2. load_gaps(events_path) -> list[dict]

EGL の `events.jsonl` を読む。1行1イベント、schema:
`{event_id, ts, run_id, event_type, object_type, object_id, payload}`。

`object_type == "KnowledgeGap"` のイベントのみ対象。同一 `object_id` に
複数イベントがある場合、**ファイル出現順で後のものが勝つ**(event sourcing)。
payload をマージ(後勝ち)して現在状態を作る。

戻り値の各要素:
```python
{"gap_id": str, "question": str, "status": str,
 "required_for": list, "is_required": bool}
```
- `is_required` は `required_for` が非空なら True(DF-2 importance の代理指標)。
- `gap_id` は payload の `gap_id`、無ければ `object_id`。
- 並び順: `is_required` True が先、その中で `gap_id` 昇順。
- パス不在なら空リスト。JSON として壊れた行は無視して数える(§5参照)。

### 3. load_backlog(path) -> list[dict]

1行1 JSON。同一 `id` は後勝ち。各要素はそのまま(最低限 `id`/`state`/`title`)。
並び順: state 優先度
`["AWAITING_TAKA","AWAITING_GPT","BLOCKED","IN_PROGRESS","OPEN","DONE"]`
の順、同順位は `id` 昇順。未知 state は最後尾。パス不在なら空リスト。

### 4. load_ledger(path) -> list[dict]

1行1 JSON の DE 台帳。フィールド名が環境で異なりうるため、
`design_evidence_id` / `de_id` / `id` / `DE` の**この順**で最初に見つかった
キーを DE 識別子とする。
どれも無い行は結果に含めず、**`ledger_unidentified` として数える**
(`unparsed` には含めない。§5/§7 参照)。

戻り値の各要素: `{"de_id": str, "record": dict}`。
並び順: `de_id` の降順(新しい DE が上)。パス不在なら空リスト。

### 5. derive_metrics(runs, gaps, backlog, ledger, unparsed, repos=None, probe=None, ledger_unidentified=0) -> dict

```python
{"runs_total": int, "runs_passed": int, "runs_needs_human": int,
 "chain_failed": int,
 "gaps_total": int, "gaps_open": int, "gaps_required_open": int,
 "backlog_awaiting": int, "backlog_done": int,
 "ledger_count": int, "unparsed_lines": int,
 "repos_total": int, "repos_dirty": int,
 "probe_checks": int, "probe_failed": int, "probe_all_ok": bool,
 "ledger_unidentified": int}
```
- `unparsed` は **JSON として解釈できない行のみ**を数えた値
  (識別子の有無は無関係)。
- `ledger_unidentified` は引数 `ledger_unidentified`(既定 0)をそのまま入れる。
- `repos` / `probe` は省略可(既定 None)。省略時は
  `repos_total=0, repos_dirty=0, probe_checks=0, probe_failed=0,
  probe_all_ok=False` とする(証拠が無い = ok ではない。fail-closed)。
- `repos_dirty` は `dirty > 0` の repo 数。
- `probe_failed` は `ok` が False の check 数、`probe_all_ok` は
  `probe["all_ok"]`。
- `runs_passed` は `status == "PASSED"` の数。
- `gaps_open` は `status == "OPEN"`、`gaps_required_open` はさらに
  `is_required` True のもの。
- `unparsed` は呼び出し側が渡す int(ローダーの戻り値はリストのみ)。
  `build` は入力 jsonl(events / backlog / ledger)を走査し、
  JSON として解釈できない行の合計を自分で数えて渡す。

### 6. render_html(metrics, runs, gaps, backlog, ledger, generated_at, repos=None, probe=None) -> str

単一 HTML 文字列。以下を**必ず含む**(テストが検査する):
```
- 文字列 "2DER" と generated_at
- 文字列 "derived"(導出物であることの明示)
- 各セクション見出し: "METRICS" / "RUNS" / "GAPS" / "BACKLOG" / "LEDGER"
- 各 run の run_id と status
- chain_ok が False の run には文字列 "CHAIN_FAIL"
- needs_human が True の run には文字列 "NEEDS_HUMAN"
- is_required かつ status=="OPEN" の gap には文字列 "REQUIRED"
- unparsed_lines > 0 のとき文字列 "UNPARSED"
- ledger_unidentified > 0 のとき文字列 "UNIDENTIFIED"
- `repos` が渡されたとき: 見出し "REPOS"、各 repo の name と head。
  `dirty > 0` の repo には文字列 "DIRTY"。
- `probe` が渡されたとき: 見出し "PROBE"、各 check の識別子と verdict。
  `ok` が False の check には文字列 "PROBE_FAIL"。
  check が 0 件のときは文字列 "NO_PROBE"。
- `repos` / `probe` が None のときは該当セクションを出さなくてよい
  (出しても構わないが、上記マーカーを誤って出さないこと)。
```
HTML エスケープを行うこと(`<` `>` `&` を含むデータで壊れない)。
デザインは自由。CSS はインラインでよい。

### 7. build(config) -> dict

```python
{"html": str, "metrics": dict}
```
`config` は dict。必須キー `runs_root` / `out_title`。
任意キー `events`(events.jsonl パス)/ `backlog` / `ledger` /
`repos`(repos.json パス)/ `probe_dir`(probe-*.jsonl のあるディレクトリ)。
必須キーが無い場合 `ViewError`。
`generated_at` は `time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime())` を使う。
上記ローダーを呼び、`unparsed` を合算して `derive_metrics` に渡し、
`render_html` の結果を返す。**ファイルには書かない。**
`repos` / `probe_dir` が指定されていれば §8 / §9 のローダーを呼び、
結果を `derive_metrics` と `render_html` に渡す。
`repos` ファイルが存在するが JSON として壊れている場合、`unparsed` に 1 を加算する。
`ledger` が指定されている場合、識別子キーを持たない行数を数えて
`derive_metrics` の `ledger_unidentified` に渡す。

### 8. load_repos(path) -> list[dict]

repo 状態は別プロセス(ハーネス層)が収集して JSON ファイルに書き出す。
本モジュールは **git を実行しない**(subprocess 禁止は維持)。読むだけ。

入力ファイル形式(JSON、jsonl ではない):
```json
{"generated_at": "2026-07-20T00:00:00Z",
 "repos": [{"name": "egl", "head": "6a4b508", "dirty": 0,
            "sync": "0/0", "error": null}]}
```
戻り値の各要素: `{"name": str, "head": str, "dirty": int, "sync": str,
"error": str|None, "is_dirty": bool}`
- 欠けたキーの既定値: `head="?"`, `dirty=-1`, `sync="?"`, `error=None`。
- `is_dirty` は `dirty > 0`。
- 並び順: `name` 昇順。
- パス不在 → 空リスト。JSON として壊れている → 空リスト(例外にしない)。
- トップレベルが dict でない、または `repos` が list でない場合も空リスト。

### 9. load_probe(probe_dir) -> dict

sandbox 封じ込め実測(4a–4f)の結果を読む。ディレクトリ内の
`probe-*.jsonl` のうち**ファイル名昇順で最後のもの**(=最新)を対象とする。
1 行 1 JSON。

戻り値:
```python
{"file": str, "checks": [{"check": str, "verdict": str, "ok": bool}],
 "all_ok": bool}
```
- check 識別子は `check` / `id` / `name` のうち最初に見つかったキーの値。
  どれも無い行は無視して数える(§7 の unparsed には含めない)。
- verdict は `verdict` / `result` / `status` のうち最初に見つかったキーの値。
  無い行は無視する。
- `ok` は `verdict` を大文字化した結果が `{"BLOCKED", "OK", "PASS"}` の
  いずれかであれば True、それ以外(LEAKED / FAIL 等)は False。
- `all_ok` は checks が 1 件以上あり、かつ全て `ok` が True のときのみ True。
  **checks が 0 件なら False**(証拠が無い = 合格ではない。fail-closed)。
- ディレクトリ不在 / 該当ファイル無し → `{"file": "", "checks": [], "all_ok": False}`。

## 禁止事項

- 外部ライブラリの import。
- ファイル書き込み・ネットワークアクセス・subprocess。
- 存在しない機構(TM-3 再検証キュー / LD-2 中心性 / IH 履歴)の表示。
  実装されていないものを UI に出さない。
- git の実行(subprocess)。repo 状態は §8 のファイルから読むだけ。
- spec にない機能の追加。
