## 2. FIX-01（run `d5f38a3f`）— なぜ止めたか

immutable 10/10 を通過し、独立再測でも 10/10。しかし `latest_balance_proof` が
契約の repo 絞込を実装していなかった。再現3事例：

| 事例 | 実装の挙動 | 仕様 |
|---|---|---|
| 自 repo proof の後に旧形式 proof | 旧形式が返る | 対象外 |
| 自 repo proof の後に他 repo proof | 他 repo が返る | 対象 repo のみ走査 |
| 他 repo の IMBALANCED が後にある | 自 repo の fresh が False | 走査対象外 |

3件とも minter ゲート(3b) に阻まれて **mint は refuse に倒れる**。false-accept 経路は
見つからなかったので breach ではない。しかし R1 が存在する理由そのものが cross-repo
状況なので、単一 repo 前提でしか正しくない実装を「R1 完了」とすると ceiling の過大主張になる。

**この便の最大の教訓**: 出荷 immutable tests は T-6/T-6b が minter 越しにしか
`latest_balance_proof` を触らず、(3b) が非適合を覆い隠した。**テスト充足 ≠ 契約充足**。
FIX-01r でこれが是正されたのは適切な処置だった。

## 4. 振動の原因は仕様側にもある（本報告の主眼）

モデルは **LBP を直すと realpath を壊し、realpath を直すと LBP が戻る**往復に入った。
その2点は、どちらも **spec 本文で固定されていない**：

1. **LBP の戻り値契約が spec に無い。**
   本文は「対象 repo の proof のみを走査」「旧形式 proof は対象外」とは書くが、
   *自 repo の proof が存在しないときに何を返すか* を書いていない。
   `(None, False)` を固定しているのは **新テストだけ**である。
   モデルは「proof は返すが fresh=False」という自然な解釈に留まり続けた。

2. **realpath 両辺適用が不変条件として明示されていない。**
   §3 の末尾に一度出るだけで、`reconcile` の分類規則の箇所には書かれていない。
   モデルは分類ロジックを書き換える際にここを落とした。

### 提案（採否は設計側）

| # | 提案 | 理由 |
|---|---|---|
| (i) | **LBP の戻り値契約を本文に明記**（自 repo の BALANCED proof が無ければ `(None, False)`） | 現状テストだけが固定しており spec 本文と齟齬 |
| (ii) | **realpath 両辺適用を「退行禁止の不変条件」として分類規則の箇所に再掲**し、T-3 positive control がその番人だと名指し | モデルが往復した2点を両方とも本文で固定する |
| (iii) | 実装アクターの見直し（現 packet は Qwen3.6-35B-A3B = 量産用。27B-Coder-Next dense FP8 `:8006` が実装用に存在） | packet の `model`/`endpoint` 変更は私の編集許可範囲（パスのみ）外につき設計側判断 |

**予算増額での押し切りは推奨しない。** 不動点が2回検出されている以上、同じ往復を
長くするだけの公算が高い。max_iterations=8 の消尽は機構としては正常動作（fail-closed）。
